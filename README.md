# Анализ пользовательских сессий (Apache Spark)

Spark-задача на Scala для анализа логов пользовательских сессий системы КонсультантПлюс.

## Описание данных

Каждый файл в директории `Сессии/` — это одна пользовательская сессия. Поддерживаемые события:

| Событие | Описание |
|---------|----------|
| `SESSION_START` / `SESSION_END` | Начало и конец сессии |
| `QS` | Быстрый поиск; следующая строка — `search_id doc1 doc2 ...` |
| `CARD_SEARCH_START` ... `CARD_SEARCH_END` | Поиск через карточку; строки параметров (`$key value`), затем строка результатов |
| `DOC_OPEN` | Открытие документа: `DOC_OPEN timestamp search_id doc_id` |

Форматы временных меток:
- `DD.MM.YYYY_HH:MM:SS`
- `DayName,_D_MonName_YYYY_HH:MM:SS_+0300`

Идентификатор документа имеет вид `<БАЗА>_<НОМЕР>`, например: `PGU_13`, `ACC_45614`, `LAW_221940`.

## Задание

1. **Задание 1** — Найти количество раз, когда в карточке поиска находили документ с идентификатором `ACC_45616`
2. **Задание 2** — Найти количество открытий каждого документа, найденного через быстрый поиск, за каждый день

## Результаты

| Задание | Результат |
|---------|-----------|
| Количество CARD_SEARCH, содержащих ACC_45616 | **479** |
| Уникальных пар (дата, документ) через QS | **75 494** |

Полные результаты сохранены в директории `output/`.

## Детали реализации

Основная логика находится в объекте:

```text
ru.hse.de2.SessionAnalyzer
```

Программа читает входные данные с помощью метода Spark:

```scala
sc.wholeTextFiles(dataPath)
```

Так как каждый файл соответствует одной пользовательской сессии, важно обрабатывать файл целиком, а не отдельные строки независимо друг от друга. Это позволяет корректно связывать события внутри одной сессии.

### Парсинг сессии

Каждый файл сессии обрабатывается функцией:

```scala
parseSession(content: String): (Seq[Seq[String]], Seq[(String, String)])
```

Функция принимает содержимое одного файла и возвращает два набора данных:

1. `Seq[Seq[String]]` — списки документов, найденных каждым поиском через карточку.
2. `Seq[(String, String)]` — пары `(date, doc_id)` для открытий документов, найденных через быстрый поиск.

Перед обработкой содержимое файла разбивается на непустые строки:

```scala
val lines = content
  .split("\n")
  .map(_.trim)
  .filter(_.nonEmpty)
  .toList
```

Далее строки последовательно обходятся с помощью индекса. Такой подход выбран потому, что одно логическое событие может занимать несколько строк. Например, после события `QS` следующая строка содержит результаты поиска, а событие `CARD_SEARCH_START` может включать несколько строк параметров до `CARD_SEARCH_END`.

### Обработка быстрого поиска QS

Для событий быстрого поиска используется структура:

```scala
mutable.Map[String, Set[String]]
```

Она хранит соответствие:

```text
search_id -> множество найденных документов
```

Когда встречается событие `QS`, программа переходит к следующей строке, где ожидаются результаты поиска:

```text
search_id doc1 doc2 ...
```

Из этой строки извлекается `search_id`, а остальные токены фильтруются как идентификаторы документов. Это позволяет позже проверить, действительно ли документ, открытый через `DOC_OPEN`, был найден именно этим быстрым поиском.

### Обработка карточки поиска

Для карточки поиска программа ищет блок:

```text
CARD_SEARCH_START
...
CARD_SEARCH_END
```

Строки параметров между началом и концом карточки пропускаются, так как для задания важен только список найденных документов. После `CARD_SEARCH_END` ожидается строка результатов:

```text
search_id doc1 doc2 ...
```

Документы из этой строки сохраняются как отдельный результат одного поиска через карточку:

```scala
cardSearchResults += docs
```

Важно, что результаты хранятся как список списков:

```scala
Seq[Seq[String]]
```

Это необходимо, потому что в первом задании нужно посчитать не количество документов `ACC_45616`, а количество поисков через карточку, в которых этот документ был найден.

### Обработка открытий документов

Событие открытия документа имеет формат:

```text
DOC_OPEN timestamp search_id doc_id
```

Для каждого `DOC_OPEN` программа:

1. извлекает дату открытия из `timestamp`;
2. извлекает `search_id`;
3. извлекает `doc_id`;
4. проверяет, что `search_id` относится к ранее найденному быстрому поиску `QS`;
5. проверяет, что `doc_id` действительно был в результатах этого `QS`.

Если все проверки успешно пройдены, сохраняется пара:

```text
date doc_id
```

Эта пара позже используется для подсчёта количества открытий каждого документа по дням.

### Обработка дат

В логах поддерживаются два формата временных меток:

```text
DD.MM.YYYY_HH:MM:SS
DayName,_D_MonName_YYYY_HH:MM:SS_+0300
```

Обе формы приводятся к единому формату:

```text
YYYY-MM-DD
```

Для этого используется функция:

```scala
parseDate(ts: String): Option[String]
```

Функция возвращает `Some(date)`, если дату удалось распознать, и `None`, если формат оказался неизвестным или повреждённым. Это позволяет программе устойчиво обрабатывать некорректные строки лога без аварийного завершения.

### Фильтрация идентификаторов документов

Для проверки идентификаторов документов используется регулярное выражение:

```scala
[A-Z]+_\d+
```

Оно соответствует идентификаторам вида:

```text
ACC_45616
LAW_221940
PGU_13
```

Фильтрация нужна для защиты от ошибочных или неожиданных строк в логах. Например, если после события поиска вместо строки результатов встретится другое событие, оно не будет ошибочно принято за документ.

### Подсчёт задания 1

После парсинга всех сессий программа получает набор результатов карточек поиска. Для первого задания выполняется следующая логика:

```scala
val cardSearchCount = parsed
  .flatMap { case (cardSearchResults, _) => cardSearchResults }
  .filter(_.contains("ACC_45616"))
  .count()
```

Здесь:

1. `flatMap` разворачивает результаты карточек поиска из всех сессий в общий поток;
2. `filter` оставляет только те поиски, где среди найденных документов есть `ACC_45616`;
3. `count` считает количество таких поисков.

### Подсчёт задания 2

Для второго задания используются пары `(date, doc_id)`, полученные из корректных событий `DOC_OPEN`, связанных с быстрым поиском `QS`.

Подсчёт выполняется так:

```scala
val qsDocOpenCounts = parsed
  .flatMap { case (_, docOpens) => docOpens }
  .map(pair => (pair, 1))
  .reduceByKey(_ + _)
  .map {
    case ((date, docId), count) =>
      s"$date\t$docId\t$count"
  }
  .sortBy(identity)
```

Здесь:

1. `flatMap` собирает все открытия документов через быстрый поиск;
2. `map(pair => (pair, 1))` превращает каждую пару `(date, doc_id)` в ключ со значением `1`;
3. `reduceByKey(_ + _)` суммирует количество одинаковых пар;
4. результат преобразуется в текстовый формат:
   ```text
   date<TAB>doc_id<TAB>count
   ```
5. строки сортируются для удобства просмотра.

### Кэширование промежуточного результата

Результат парсинга сессий используется дважды: для первого и второго задания. Поэтому после парсинга применяется кэширование:

```scala
.cache()
```

Это позволяет Spark не читать и не парсить входные файлы повторно при вычислении второй задачи.
